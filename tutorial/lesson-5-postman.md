## 🧠 ทำไมต้องทดสอบ API ก่อน?

ก่อนเขียนโค้ดเชื่อมต่อ Frontend กับ Backend มาเข้าใจภาพรวมก่อน:

> 💡 **เปรียบเทียบ:**
>
> ```
> ┌──────────────┐         ┌──────────────┐         ┌──────────────┐
> │   Frontend   │  ────>  │     API      │  ────>  │   Database   │
> │   (Vue.js)   │  <────  │   (Django)   │  <────  │   (SQLite)   │
> └──────────────┘         └──────────────┘         └──────────────┘
>       หน้าเว็บ              พนักงานเสิร์ฟ              ห้องครัว
> ```
>
> | ถ้าเป็นร้านอาหาร | ถ้าเป็นแอป |
> |------------------|------------|
> | ลูกค้า (เรา) | ผู้ใช้หน้าเว็บ (Frontend) |
> | เมนูอาหาร | API Endpoints (URL ต่างๆ) |
> | พนักงานเสิร์ฟ | Backend (Django) |
> | ห้องครัว | Database (SQLite) |
> | ใบสั่งอาหาร | HTTP Request |
> | อาหารที่ได้ | HTTP Response (JSON) |
>
> **Postman = เหมือนเราไปสั่งอาหารเองที่เคาน์เตอร์** โดยไม่ต้องผ่านหน้าเว็บ

## 🧠 HTTP Methods คืออะไร?

HTTP Methods คือ **วิธีที่ Frontend บอก Backend ว่าต้องการทำอะไร:**

| HTTP Method | ทำอะไร | เปรียบเทียบ | CRUD |
|------------|--------|-------------|------|
| `GET` | **ขอดู**ข้อมูล | 📖 เปิดดูเมนู | **R**ead |
| `POST` | **สร้าง**ข้อมูลใหม่ | 📝 เขียนใบสั่งอาหาร | **C**reate |
| `PUT` | **แก้ไข**ข้อมูลที่มีอยู่ | ✏️ แก้ไบสั่ง "เปลี่ยนจากข้าวผัดเป็นผัดไทย" | **U**pdate |
| `DELETE` | **ลบ**ข้อมูล | 🗑️ ยกเลิกออเดอร์ | **D**elete |

> 💡 **CRUD** (Create, Read, Update, Delete) คือ 4 ปฏิบัติการพื้นฐานที่ใช้กับข้อมูลทุกชนิด
> แอปส่วนใหญ่ในโลกก็มีแค่ 4 อย่างนี้!

## ส่วนที่ 2: รู้จัก API Endpoints

### ขั้นตอนที่ 2: ทำความเข้าใจ API ของ Note App

Backend เตรียม API ไว้ให้แล้ว 5 เส้นทาง:

| # | Method | URL | คำอธิบาย | ต้องส่ง Body? |
|---|--------|-----|----------|---------------|
| 1 | `GET` | `/api/notes/` | ดึงรายการโน้ตทั้งหมด | ❌ ไม่ต้อง |
| 2 | `POST` | `/api/notes/create/` | สร้างโน้ตใหม่ | ✅ ต้องส่ง |
| 3 | `GET` | `/api/notes/{id}/` | ดึงข้อมูลโน้ต 1 รายการ | ❌ ไม่ต้อง |
| 4 | `PUT` | `/api/notes/{id}/update/` | แก้ไขโน้ต | ✅ ต้องส่ง |
| 5 | `DELETE` | `/api/notes/{id}/delete/` | ลบโน้ต | ❌ ไม่ต้อง |

> 💡 `{id}` คือ **ตัวเลข** ที่แทน id ของโน้ต เช่น `/api/notes/1/`, `/api/notes/42/`

**โครงสร้างข้อมูล Note:**

```json
{
  "id": 1,
  "title": "ซื้อของ",
  "body": "ไข่ นม ขนมปัง",
  "is_done": false,
  "created_at": "2026-04-07T10:00:00Z",
  "updated_at": "2026-04-07T10:00:00Z"
}
```

| Field | ประเภท | คำอธิบาย | ต้องส่งตอนสร้าง? |
|-------|--------|----------|------------------|
| `id` | ตัวเลข | ID อัตโนมัติ | ❌ ระบบสร้างให้ |
| `title` | ข้อความ | ชื่อโน้ต (ต้องกรอก) | ✅ **ต้องส่ง** |
| `body` | ข้อความ | เนื้อหาโน้ต (ว่างได้) | ⭕ ไม่ส่งก็ได้ (default = `""`) |
| `is_done` | true/false | ทำเสร็จหรือยัง | ⭕ ไม่ส่งก็ได้ (default = `false`) |
| `created_at` | วันเวลา | วันที่สร้าง | ❌ ระบบสร้างให้ |
| `updated_at` | วันเวลา | วันที่แก้ไขล่าสุด | ❌ ระบบสร้างให้ |

#### ✅ ตรวจสอบ
- [ ] เข้าใจว่ามี API 5 เส้นทาง
- [ ] เข้าใจว่า `title` ต้องกรอก แต่ `body` กับ `is_done` ไม่บังคับ
- [ ] เข้าใจว่า `id`, `created_at`, `updated_at` ถูกสร้างโดยระบบอัตโนมัติ

### ขั้นตอนที่ 4: ทดสอบ GET — ดูรายการโน้ตทั้งหมด

นี่คือ API แรกที่เราจะทดสอบ — ขอดูโน้ตทั้งหมด

**ใน Postman:**
1. เลือก Method: **GET**
2. ใส่ URL: `http://localhost:8000/api/notes/`
3. กดปุ่ม **Send**

**ผลลัพธ์ที่ควรได้:**

```json
[]
```

> 💡 ได้ `[]` (array ว่าง) เพราะยังไม่มีโน้ตในฐานข้อมูล — ถูกต้องแล้ว!
>
> | ส่วน | ค่า | ความหมาย |
> |------|-----|----------|
> | Status | `200 OK` | สำเร็จ! ✅ |
> | Body | `[]` | ไม่มีข้อมูล (array ว่าง) |

#### ✅ ตรวจสอบ

### ขั้นตอนที่ 5: ทดสอบ POST — สร้างโน้ตใหม่

ตอนนี้มาลองสร้างโน้ตใหม่กัน!

**ใน Postman:**
1. เลือก Method: **POST**
2. ใส่ URL: `http://localhost:8000/api/notes/create/`
3. กดแท็บ **Body**
4. เลือก **raw**
5. เปลี่ยน type จาก `Text` เป็น **JSON** (dropdown ด้านขวา)
6. พิมพ์ข้อมูลนี้:

```json
{
  "title": "ซื้อของ",
  "body": "ไข่ นม ขนมปัง"
}
```

7. กด **Send**

**ผลลัพธ์ที่ควรได้:**

```json
{
  "id": 1,
  "title": "ซื้อของ",
  "body": "ไข่ นม ขนมปัง",
  "is_done": false,
  "created_at": "2026-04-07T10:05:00Z",
  "updated_at": "2026-04-07T10:05:00Z"
}
```

> 💡 **สิ่งที่ต้องสังเกต:**
>
> | สิ่งที่สังเกต | ค่า | ความหมาย |
> |--------------|-----|----------|
> | Status | `201 Created` | สร้างสำเร็จ! ✅ |
> | `id` | `1` | ระบบสร้าง id อัตโนมัติ |
> | `is_done` | `false` | เราไม่ได้ส่ง → ใช้ค่า default |
> | `created_at` | วันเวลาปัจจุบัน | ระบบบันทึกอัตโนมัติ |
>
> 🤔 **ทำไม Status เป็น `201` ไม่ใช่ `200`?**
>
> | Status Code | ความหมาย | ใช้เมื่อไหร่ |
> |-------------|----------|-------------|
> | `200 OK` | สำเร็จ (ทั่วไป) | GET, PUT |
> | `201 Created` | สร้างข้อมูลใหม่สำเร็จ | POST |
> | `204 No Content` | สำเร็จ ไม่มีข้อมูลส่งกลับ | DELETE |
> | `400 Bad Request` | ส่งข้อมูลผิด | ส่ง body ไม่ถูก format |
> | `404 Not Found` | ไม่พบข้อมูล | ไม่มี id ที่ขอ |

**ลองสร้างเพิ่มอีก 2 รายการ:**

โน้ตที่ 2:
```json
{
  "title": "การบ้าน",
  "body": "ทำแบบฝึกหัดบทที่ 5"
}
```

โน้ตที่ 3:
```json
{
  "title": "ประชุม",
  "body": "ประชุมทีมบ่าย 2 โมง",
  "is_done": true
}
```

> 💡 สังเกตว่าโน้ตที่ 3 เราส่ง `"is_done": true` → แบบนี้ก็ได้!

#### ✅ ตรวจสอบ
- [ ] สร้างโน้ต 3 รายการสำเร็จ (ทุกรายการได้ Status `201`)
- [ ] แต่ละรายการมี `id` ที่ต่างกัน (1, 2, 3)
- [ ] เข้าใจว่าต้องเลือก Body → raw → JSON แล้วพิมพ์ JSON
- [ ] เข้าใจ Status Code ต่างๆ (200, 201, 400, 404)

### ขั้นตอนที่ 8: ทดสอบ GET + search — ค้นหาโน้ต

API รองรับการค้นหาด้วย **query parameter** `?search=`:

**ใน Postman:**
1. เลือก Method: **GET**
2. ใส่ URL: `http://localhost:8000/api/notes/?search=ซื้อ`
3. กด **Send**

**ผลลัพธ์:**

```json
[
  {
    "id": 1,
    "title": "ซื้อของ",
    "body": "ไข่ นม ขนมปัง",
    "is_done": false,
    "created_at": "...",
    "updated_at": "..."
  }
]
```

> 💡 **Query Parameter คืออะไร?**
>
> ```
> http://localhost:8000/api/notes/?search=ซื้อ
>                                 ↑
>                            ? = เริ่มต้น query parameter
>                            search = ชื่อ parameter
>                            ซื้อ = ค่าที่ค้นหา
> ```
>
> ใน Postman ยังสามารถใส่ผ่านแท็บ **Params** ได้:
>
> | Key | Value |
> |-----|-------|
> | search | ซื้อ |

**ลองค้นหาอื่นๆ:**
- `?search=ประชุม` → ได้โน้ต "ประชุม"
- `?search=xyz` → ได้ `[]` (ไม่พบ)
- ไม่ใส่ `?search=` → ได้ทุกรายการ

#### ✅ ตรวจสอบ
- [ ] ค้นหาโน้ตด้วย `?search=` สำเร็จ
- [ ] เข้าใจว่า Query Parameter ใส่หลังเครื่องหมาย `?` ใน URL

### ขั้นตอนที่ 9: ทดสอบ PUT — แก้ไขโน้ต

มาลองแก้ไขโน้ต id = 1 จาก "ซื้อของ" เป็น "ซื้อของ (แก้ไขแล้ว)":

**ใน Postman:**
1. เลือก Method: **PUT**
2. ใส่ URL: `http://localhost:8000/api/notes/1/update/`
3. กดแท็บ **Body** → เลือก **raw** → เปลี่ยนเป็น **JSON**
4. พิมพ์:

```json
{
  "title": "ซื้อของ (แก้ไขแล้ว)",
  "body": "ไข่ นม ขนมปัง เนย ชีส",
  "is_done": true
}
```

5. กด **Send**

**ผลลัพธ์:**

```json
{
  "id": 1,
  "title": "ซื้อของ (แก้ไขแล้ว)",
  "body": "ไข่ นม ขนมปัง เนย ชีส",
  "is_done": true,
  "created_at": "...",
  "updated_at": "..."
}
```

> 💡 **สิ่งที่สังเกต:**
>
> | สิ่งที่เปลี่ยน | ก่อน | หลัง |
> |--------------|------|------|
> | `title` | ซื้อของ | ซื้อของ (แก้ไขแล้ว) |
> | `body` | ไข่ นม ขนมปัง | ไข่ นม ขนมปัง เนย ชีส |
> | `is_done` | false | true |
> | `updated_at` | เวลาเดิม | **อัปเดตเป็นเวลาใหม่** |
> | `created_at` | เวลาเดิม | **ไม่เปลี่ยน** (บันทึกแค่ครั้งแรก) |

#### ✅ ตรวจสอบ
- [ ] แก้ไขโน้ต id = 1 สำเร็จ (Status `200`)
- [ ] `updated_at` เปลี่ยน แต่ `created_at` ไม่เปลี่ยน
- [ ] ลอง GET `/api/notes/1/` อีกครั้ง → เห็นข้อมูลที่แก้ไขแล้ว

### ขั้นตอนที่ 10: ทดสอบ DELETE — ลบโน้ต

มาลองลบโน้ต id = 1:

**ใน Postman:**
1. เลือก Method: **DELETE**
2. ใส่ URL: `http://localhost:8000/api/notes/1/delete/`
3. **ไม่ต้อง**ส่ง Body
4. กด **Send**

**ผลลัพธ์:**
- Status: **`204 No Content`**
- Body: ว่างเปล่า (ไม่มีข้อมูลส่งกลับ)

> 💡 `204 No Content` หมายถึง "สำเร็จ แต่ไม่มีอะไรจะส่งกลับ" — เพราะข้อมูลถูกลบไปแล้ว!

**ทดสอบว่าลบจริง:**
- กด GET `http://localhost:8000/api/notes/` → ควรเหลือ 2 รายการ (ไม่มี id = 1 แล้ว)
- กด GET `http://localhost:8000/api/notes/1/` → ได้ `404 Not Found`

#### ✅ ตรวจสอบ
- [ ] ลบโน้ต id = 1 สำเร็จ (Status `204`)
- [ ] GET รายการทั้งหมด → เหลือ 2 รายการ
- [ ] GET โน้ต id = 1 → ได้ `404 Not Found`

### ขั้นตอนที่ 11: ทดสอบ Error — ส่งข้อมูลผิด

มาดูว่า Backend จัดการข้อมูลผิดยังไง:

**ทดสอบที่ 1 — สร้างโน้ตโดยไม่ส่ง title:**

1. POST `http://localhost:8000/api/notes/create/`
2. Body:
```json
{
  "body": "เนื้อหาอย่างเดียว"
}
```
3. กด Send

**ผลลัพธ์:**
```json
{
  "title": ["This field is required."]
}
```
- Status: **`400 Bad Request`** — ข้อมูลไม่ถูกต้อง!

---

**ทดสอบที่ 2 — ส่ง title ว่าง:**

```json
{
  "title": ""
}
```

**ผลลัพธ์:**
```json
{
  "title": ["This field may not be blank."]
}
```

---

**ทดสอบที่ 3 — แก้ไขโน้ตที่ไม่มี:**

1. PUT `http://localhost:8000/api/notes/999/update/`
2. Body:
```json
{
  "title": "ทดสอบ"
}
```

**ผลลัพธ์:**
```json
{
  "detail": "Not found."
}
```
- Status: **`404 Not Found`**

> 💡 **ทำไมต้องทดสอบ Error?**
>
> เมื่อเราเขียนโค้ด Frontend ในบทถัดไป เราจะต้อง **จัดการ error เหล่านี้** ให้ผู้ใช้เห็นข้อความที่เข้าใจได้
> ถ้ารู้ว่า Backend ตอบ error มาแบบไหน เราจะเขียนโค้ดจัดการได้ง่ายขึ้น!

#### ✅ ตรวจสอบ
- [ ] ลองส่งข้อมูลผิด → ได้ Status `400`
- [ ] เข้าใจว่า error message บอกว่าอะไรผิด
- [ ] ลองเข้าถึง id ที่ไม่มี → ได้ Status `404`

| # | Method | URL | Body | Status สำเร็จ | ทำอะไร |
|---|--------|-----|------|------------|--------|
| 1 | GET | `/api/notes/` | ❌ | 200 | ดูรายการทั้งหมด |
| 2 | GET | `/api/notes/?search=คำ` | ❌ | 200 | ค้นหาตาม title |

| 4 | GET | `/api/notes/{id}/` | ❌ | 200 | ดูโน้ต 1 รายการ |
| 5 | PUT | `/api/notes/{id}/update/` | ✅ JSON | 200 | แก้ไขโน้ต |
| 6 | DELETE | `/api/notes/{id}/delete/` | ❌ | 204 | ลบโน้ต |

## ❌ ปัญหาที่พบบ่อย

### ปัญหา: กด Send แล้วได้ `Could not send request` หรือ `Error: connect ECONNREFUSED`

**สาเหตุ:** Django Backend ไม่ได้ทำงานอยู่

**วิธีแก้:**
1. ตรวจสอบว่ารัน `uv run python manage.py runserver` อยู่
2. ตรวจว่า Terminal ไม่มี error
3. ตรวจว่า URL ใน Postman ถูกต้อง — ต้องเป็น `http://localhost:8000/api/notes/` (**ไม่ใช่** `localhost:5173`)

---

### ปัญหา: POST แล้วได้ `415 Unsupported Media Type`

**สาเหตุ:** ไม่ได้เลือก Body type เป็น JSON

**วิธีแก้:**
1. กดแท็บ **Body**
2. เลือก **raw**
3. เปลี่ยน dropdown ด้านขวาจาก `Text` → **JSON**

> ⚠️ ถ้าเลือก `Text` แทน `JSON` → Postman จะไม่ส่ง header `Content-Type: application/json` ทำให้ Backend ไม่เข้าใจข้อมูล

---

### ปัญหา: POST แล้วได้ `400 Bad Request` — `"Expecting value: line 1 column 1"`

**สาเหตุ:** JSON format ผิด เช่น ลืมเครื่องหมาย `"` หรือใส่ comma เกิน

**วิธีแก้:** ตรวจสอบ JSON ให้ถูกต้อง:

```json
// ❌ ผิด — ลืม " ปิด
{ "title": ซื้อของ }

// ❌ ผิด — มี comma ตัวสุดท้าย (JSON ไม่อนุญาต!)
{ "title": "ซื้อของ", }

// ❌ ผิด — ใช้ single quote
{ 'title': 'ซื้อของ' }

// ✅ ถูก
{ "title": "ซื้อของ" }
```

---

### ปัญหา: URL ลืมใส่ `/` ตัวสุดท้าย

**สาเหตุ:** Django ต้องการ `/` ท้าย URL (trailing slash)

**วิธีแก้:**
```
❌  http://localhost:8000/api/notes         ← ขาด /
✅  http://localhost:8000/api/notes/        ← ถูกต้อง

❌  http://localhost:8000/api/notes/create  ← ขาด /
✅  http://localhost:8000/api/notes/create/ ← ถูกต้อง
```

### ⭐ ระดับง่าย
สร้างโน้ตเพิ่มอีก 3 รายการ แล้วลองค้นหาด้วย `?search=` ให้ค้นเจอทีละรายการ

### ⭐⭐ ระดับปานกลาง
ลองทำ CRUD ครบวงจร:
1. สร้างโน้ตใหม่ (POST)
2. ดูข้อมูลที่สร้าง (GET by id)
3. แก้ไขข้อมูล (PUT)
4. ดูข้อมูลที่แก้ไขแล้ว (GET by id — ตรวจว่าเปลี่ยนจริง)

- ลองกดปุ่ม POST ในหน้า Browsable API
- สังเกตว่าหน้าตาต่างจาก Postman ยังไง
- ลองเข้า `/api/notes/1/` ในเบราว์เซอร์

## 📖 คำศัพท์ที่เรียนรู้ในบทนี้

| คำศัพท์ | ความหมาย | เปรียบเทียบ |
|---------|----------|-------------|
| API | Application Programming Interface — ช่องทางที่ Frontend คุยกับ Backend | เหมือนเมนูร้านอาหาร — สั่งได้ตามที่มี |
| REST API | รูปแบบ API ที่ใช้ HTTP Methods + URL จัดการข้อมูล | เหมือนระบบไปรษณีย์ — มีที่อยู่ (URL) + ประเภทบริการ (GET/POST/PUT/DELETE) |
| HTTP Method | วิธีที่บอก Backend ว่าต้องการทำอะไร (GET, POST, PUT, DELETE) | เหมือนคำกริยา — ดู, สร้าง, แก้ไข, ลบ |
| Status Code | ตัวเลขที่ Backend ส่งกลับบอกว่าสำเร็จหรือผิดพลาด | เหมือนป้ายตอบกลับ — 200 = ได้ครับ, 404 = ไม่มีครับ |
| Request | ข้อมูลที่ส่งไป Backend | เหมือนใบสั่งอาหาร |
| Response | ข้อมูลที่ Backend ส่งกลับมา | เหมือนอาหารที่เสิร์ฟ |
| JSON | JavaScript Object Notation — รูปแบบข้อมูลที่ใช้ส่งระหว่าง Frontend กับ Backend | เหมือนภาษากลาง — ทั้ง Frontend และ Backend อ่านรู้เรื่อง |
| Query Parameter | ตัวแปรใน URL หลังเครื่องหมาย `?` | เหมือนตัวเลือกเพิ่มเติม — "ขอเมนูที่มี 'ไก่'" |
| Endpoint | URL + Method ที่เฉพาะเจาะจง เช่น `GET /api/notes/` | เหมือนเคาน์เตอร์บริการ — เคาน์เตอร์ 1 รับสั่ง, เคาน์เตอร์ 2 รับแก้ไข |
| CRUD | Create, Read, Update, Delete — 4 ปฏิบัติการพื้นฐาน | เหมือนสมุดบันทึก — เขียน, อ่าน, แก้, ลบ |
| Postman | เครื่องมือสำหรับทดสอบ API โดยไม่ต้องเขียนโค้ด | เหมือนรีโมท — กดปุ่มแล้วดูว่า Backend ตอบอะไร |

## สรุปสิ่งที่ได้เรียนรู้ ✅

| หัวข้อ | สิ่งที่เรียนรู้ |
|--------|----------------|
| API | ช่องทางที่ Frontend คุยกับ Backend ผ่าน HTTP |
| REST API | ใช้ HTTP Methods (GET, POST, PUT, DELETE) + URL จัดการข้อมูล |
| HTTP Methods | GET = ดู, POST = สร้าง, PUT = แก้ไข, DELETE = ลบ |
| Status Code | 200 = OK, 201 = Created, 204 = No Content, 400 = Bad Request, 404 = Not Found |
| Postman | เครื่องมือทดสอบ API — ใส่ Method + URL + Body → กด Send |
| JSON | รูปแบบข้อมูลที่ใช้ส่งระหว่าง Frontend/Backend |
| Query Parameter | ใส่หลัง `?` ใน URL เช่น `?search=คำค้นหา` |
| Error Handling | Backend ตอบ error message + Status Code ที่ชัดเจน |
